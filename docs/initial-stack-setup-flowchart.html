Initial Stack Setup and Creation Flow

This shows how a clean Proxmox VE node becomes a complete ProxmoxVEAutopilot stack: PVE host setup, Ubuntu controller VM, Docker runtime, Windows build host, source-built artifacts, promoted ISOs, and operational readiness.

Customer Guide Windows Deployment Flow

High-Level Mental Model

Initial setup has four machines or roles: your operator workstation, the Proxmox VE host, the Ubuntu controller VM, and the Windows build host. The Proxmox host is only the bootstrapper; the long-running Autopilot app runs in the Ubuntu controller.

1. Run the installerThe Proxmox console installer is the main setup driver. It calls the lower-level phases for you.
2. FoundationThe script repairs PVE API access, storage, secrets, and creates or discovers the Ubuntu controller VM.
3. Controller runtimeThe script syncs source to the controller and starts Docker Compose for web, MCP, Postgres, monitor, and builder.
4. Bootstrap mediaThe script finds or downloads Windows and VirtIO media and publishes readiness to /setup.
5. Build hostThe controller, triggered by setup, creates or repairs a Windows build-host VM and queues source-build workloads.
6. OperationalThe script promotes artifacts to PVE ISO storage and /setup becomes ready for real Windows provisioning.
Main point: the operator mostly runs one console installer on the Proxmox host. That script drives foundation, bootstrap, controller sync, media readiness, operational repair, and setup-state publishing. The controller and build host do their jobs because the script gets the stack into the right state.
Foundation Make PVE usable for Autopilot: repair apt enough for shell tools, create API token/role/ACLs, detect storage/bridge/node/IP, generate secrets, create or find the Ubuntu controller VM, sync source, start controller runtime, verify /healthz.
Bootstrap Make deployment inputs ready: revalidate and sync the PVE token/config, find or download Windows and VirtIO media, publish media readiness, detect build-host state, and create/repair blank deployment templates.
Operational Make the stack usable: verify controller health, promote source-built setup artifacts, pull large ISOs into PVE storage when needed, republish setup state, and mark /setup ready for real Windows provisioning.

Merged Setup Flow

Read this as one lifecycle. The setup script is idempotent, so Foundation, Bootstrap, and Operational can be rerun to repair drift without moving Docker or Windows build tooling onto the Proxmox host.

PVE host: bootstrapper only. It owns API token/ACL repair, storage/media scanning, controller VM creation, artifact transfer targets, and template VM repair.
Ubuntu controller: long-running app home. It owns Docker Compose, web UI/API, MCP, Postgres, monitoring, setup state, build-host orchestration, and artifact promotion.
Windows build host: disposable source-builder. It owns ADK, WinPE add-on, .NET, WiX, MSI builds, WinPE builds, and CloudOSD builds. PVE never gets these tools.
