Customer Guide Initial Setup Flow

High-Level Mental Model

Think of the software as a Windows deployment factory. The controller prepares media, Proxmox boots a blank VM into that media, WinPE lays down Windows, then the installed OS phones home so the controller can finish identity, Autopilot, domain, and role work.

1. Pick a pathDesktop client uses OSDCloud. Windows Server uses OSDeploy.
2. Prepare mediaBuild or reuse a bootable ISO, publish it to Proxmox, and verify hashes.
3. Create a runPreflight checks pass, then the controller creates the run and task sequence.
4. Boot WinPEProxmox clones a blank VM, attaches the ISO, and WinPE calls back to the controller.
5. Lay down WindowsWinPE installs Windows, stages drivers, SetupComplete, OSD client, and AutopilotAgent.
6. Finish in full OSThe installed OS heartbeats, then optional hash, domain, and server role steps run.

OSDCloud Desktop

Use this for client Windows. OSDCloud handles the Windows deployment. ProxmoxVEAutopilot wraps it with VM creation, cached download support, first-boot staging, AutopilotAgent, optional Autopilot hash upload, optional AD domain verification, and readiness evidence.

OSDeploy Server

Use this for Windows Server. OSDeploy builds and applies a Server image. After the base OS is up, Task Sequence v2 can continue into domain join, file server, isolated domain controller, MECM prereq, or Lab in a Box role work.

The gray nodes are shared concepts, green is OSDCloud desktop, blue is OSDeploy server, purple is Task Sequence v2, cyan is Autopilot/cloud evidence, yellow is a gate, and red is a stop/failure state.

CloudOSD: desktop client flow. OSDCloud applies Windows, then CloudOSD stages AutopilotAgent, optional domain join unattend, optional hash capture/upload, and readiness evidence.
OSDeploy: server flow. OSDeploy applies the Server image, then Task Sequence v2 handles base readiness and optional full-OS role automation.
Task Sequence v2: the controller creates the plan at run creation; AutopilotAgent claims work in full Windows and reports logs/results back to /osd/v2.
