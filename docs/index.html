What This Platform Does

Proxmox VE Autopilot turns a clean Proxmox VE node into a browser-managed Windows deployment platform. The Proxmox host stays focused on virtualization while the Ubuntu controller VM owns the web UI, API, Postgres, monitor, builder, MCP service, and setup state.

Main customer takeaway: most setup starts with one console installer on the Proxmox VE host. The script prepares Proxmox access, creates the controller, starts the stack, handles media gates, coordinates the Windows build host, promotes artifacts, and publishes readiness into the web UI.
Primary output Operational Windows deployment factory

Operators choose a deployable OS path instead of caring about low-level artifacts.

Proxmox VE Host

Runs the installer, provides VM/storage/networking, hosts ISO media, and boots deployment targets.

Ubuntu Controller

Runs the application stack, setup state, APIs, Postgres, monitor, builder orchestration, and evidence collection.

Windows Build Host

Builds Windows-only outputs: AutopilotAgent MSI, WinPE, OSDCloud media, and OSDeploy media.

Operator Browser

Drives setup, artifact promotion, deployment runs, readiness review, and recovery workflows.

Initial Setup: Foundation -> Bootstrap -> Operational

The console installer performs the stack creation and moves the environment through three customer-visible phases.

Foundation

Prepares Proxmox and creates or discovers the controller.

  • API token, role, and ACLs
  • Storage, bridge, node, and IP detection
  • Ubuntu controller VM
  • Controller runtime health

Bootstrap

Prepares deployment inputs and setup state.

  • Windows installation media
  • VirtIO driver media
  • Setup state publishing
  • Blank deployment templates

Operational

Makes the stack usable for operators.

  • Controller health verified
  • Build artifacts promoted
  • ISO storage updated
  • /setup reports ready

Windows Deployment Workflows

After setup is ready, users choose a path based on whether they are deploying workstations or servers. Both paths are tracked as Task Sequence v2 runs and finish with readiness evidence.

Workstations

Client Windows deployments use OSDCloud.

  • Client Windows image deployment
  • AutopilotAgent first-boot setup
  • Optional Autopilot hardware hash capture with custom group tag
  • Optional Intune management
  • Optional domain join
  • 33 built-in virtual device models across 7 manufacturers
  • 16 device families and 6 chassis types for VM identity

Servers

Windows Server deployments use OSDeploy.

  • Server image deployment
  • Base readiness verification
  • Optional domain join
  • File server
  • Isolated domain controller
  • MECM prereq and/or basic setup with clients connected
  • Lab in a Box
  • Firewall and isolated subnet support
1. Choose pathWorkstation or server
2. PreflightArtifact, storage, media, role checks
3. Create runTask Sequence v2 plan created
4. Boot WinPEVM registers with controller
5. Install WindowsDrivers, setup, OSD client, agent
6. FinishHeartbeat, optional hash/domain/role work

Operational Readiness

Readiness is based on evidence, not assumptions. The platform marks setup or deployment ready only after the required health, artifact, heartbeat, and role checks have been observed.

Stack Ready

Controller healthy, services running, media ready, build host fresh, artifacts promoted.

Workstation Ready

OSDCloud completed, Windows booted, AutopilotAgent heartbeated, optional hash/domain evidence present.

Server Ready

Server image applied, Windows booted, AutopilotAgent online, QGA known, role status ready.

Artifact Ready

Built from source, uploaded with metadata, hashed, and promoted to Proxmox ISO storage.

Open Platform

Proxmox VE Autopilot is designed to run inside the customer's environment with visible source, local control, and no hosted product dependency.

Built from source: runtime binaries in the stack, including AutopilotAgent, are built from source instead of being supplied through a vendor-controlled build pipeline.
Code signing: supported and recommended for production environments, but not required to deploy the stack. Planned trust options include Intune Cloud PKI and Azure Trusted Signing.

No Vendor Telemetry

No hosted product analytics, SaaS tracking, or background call-home path. Deployment heartbeats and readiness evidence stay on the customer controller.

MIT Licensed

Permissive open-source licensing for use, modification, redistribution, and customer-owned extensions.

Self-Hosted Control

The control plane runs on the customer's Proxmox, Ubuntu controller, Windows build host, storage, network, and identity systems.

Source-Visible Workflows

Installer scripts, OSDCloud, OSDeploy, WinPE, task sequencing, and artifact promotion are inspectable and operator-owned.
